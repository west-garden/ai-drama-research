# 对自建系统的架构启发

从调研中提炼的、对构建 Agent 驱动自进化系统有参考价值的设计思路。

## 来自巨日禄调研的架构启发

### 1. 模型路由层 — 核心架构组件

巨日禄最重要的架构思路是**不同场景用不同模型**：叙事用星河、打斗用星辰、细节用星澜、真人用星光。自建系统应该把"模型路由"作为核心中间层：

```
Agent 分析镜头类型 → 模型路由器 → 选择最优商业模型 API → 生成
```

路由策略应支持：
- 按镜头类型（叙事/打斗/特写/特效）自动选模型
- 按质量要求（效率优先 vs 质量优先）切换模式
- 新模型上线后快速接入，不改动上层逻辑

### 2. 双工作流模式设计

巨日禄的两种 Agent 模式值得借鉴：
- **批量优先模式**（Agent 融生模式）：适合日常叙事，一次批量跑完再补镜头
- **质量优先模式**（Agent 图生模式）：适合复杂镜头，每个镜头单独调模型

自建系统应支持类似的模式切换，让 Agent 根据内容类型自动决定用哪种流水线。

### 3. 融图思路 — 角色一致性的工程方案

不等模型本身解决一致性问题，而是通过**参考图 + 场景 + 姿态**合成来保证。这是一个务实的工程方案。自建系统的角色管理模块应该：
- 维护角色资产库（三视图/参考图）
- 每次生成时注入角色参考
- 支持多角色空间关系控制

### 4. 结构化提示词自动生成

巨日禄花大量文档教用户写提示词（景别+氛围+运镜+视角+构图+风格+动态控制）。在 Agent 系统中，这应该完全自动化：

```
剧本文本 → Agent 分析 → 自动生成结构化提示词 {
  景别: "近景",
  氛围: "冷色调阴暗",
  运镜: "缓慢推进",
  动态: "小幅动态",
  ...
}
```

### 5. 分镜策略库的标签匹配与选择机制

来自火山剧创「200+ 爆款策略」的工程启示：策略库不应是黑盒描述，而应是**带多维度标签的结构化数据**，由规则引擎做匹配，大模型只做标签生成和最终微调。

**匹配流程：**
1. **剧本解析 Agent 输出标签**：场景类型、情绪弧、权力关系、叙事节拍、节奏、人数、空间
2. **策略条目带元数据**：每个策略有 `tags`（适用场景）+ `shot_formula`（镜头公式）+ `priority`
3. **硬匹配筛选**：`narrative_beat` + `emotion_arc` 从 200 个里筛出 8-15 个候选
4. **软匹配精筛**：用 `setting`、`pacing`、`character_count` 缩到 3-5 个
5. **连贯性校验 Agent**：检查相邻场景是否重复同类运镜，调整优先级防止审美疲劳
6. **最终注入**：选中策略的 `shot_formula` 自动填入结构化提示词

**关键原则：** 大模型不适合在 200 个长文本里做精确选择，精确匹配交给结构化规则。

### 6. 项目数据结构

巨日禄的三级组织值得参考：

```
项目（一部剧）
  └── 集（Episode）
       └── 片段/分镜（Shot）
            ├── 角色引用
            ├── 场景描述
            ├── 结构化提示词
            ├── 生成结果（图片/视频）
            └── 状态（待生成/已生成/已审核）
```

### 6. 架构必须支持快速模型切换

巨日禄的经验证明模型花期不超过两个月，周级迭代是常态。自建系统的架构底线：
- 模型接入层要做抽象，新模型接入不应改动业务逻辑
- 支持 A/B 测试不同模型在同一镜头上的效果
- 模型配置应可热更新

### 7. Agent 应补齐的环节

巨日禄证明了"融图+视频生成"这条路可行，但刻意不做的环节恰好是 Agent 系统的差异化空间：

| 环节 | 巨日禄 | Agent 系统应做 |
|------|--------|---------------|
| 剧本生成 | ❌ | LLM 自动生成+迭代优化 |
| 分镜拆解 | ✅ 半自动 | 全自动，Agent 理解叙事结构 |
| 提示词 | ✅ 需人工调 | Agent 自动生成结构化提示词 |
| 画面生成 | ✅ | 模型路由 + 商业 API |
| 视频生成 | ✅ | 同上 |
| 配音 | ❌ | 集成 TTS API |
| 口型同步 | ❌ | 集成 lip-sync 模型 |
| 字幕 | ❌ | 自动生成 |
| 质量评估 | ❌ 人工 | Agent 自动评估+重试（参见 [质量基准体系](quality-benchmark.md)） |
| 爆款预测 | ❌ | 基于数据分析的内容评分 |

### 8. 成本控制要从架构层面考虑

巨日禄的成本演进（5000→400 元/分钟）说明成本是竞争的核心维度。自建系统需要：
- 每次 API 调用的成本跟踪
- 质量/成本平衡的自动决策（不是所有镜头都需要最贵的模型）
- 生成失败时的重试策略（避免浪费 token）

## 来自 LibTV 调研的架构启发

> LibTV 是 LiblibAI 推出的通用 AI 视频创作平台，不是漫剧垂直工具。但它的**双入口架构、Skill 中间层、Agent 状态管理**对自建系统有重要参考价值。详细调研见 [tools/libtv.md](../tools/libtv.md)。

### 9. 双入口架构 — 从第一天起为 Agent 设计

LibTV 最根本的架构思路是：**人类和 Agent 从产品设计第一天就是并行的入口**，不是"先做 GUI 再补 API"。

```
传统模式：
  GUI 产品 → 后补 API → Agent 模拟人类操作

LibTV 模式：
  共享能力层（模型、工作流、项目管理）
      ├── 创作者入口：GUI 画布（针对人类认知优化）
      └── Agent 入口：Skill 接口（针对 Agent 执行优化）
```

**对自建系统的启示：**

自建系统不应该先做一个人类用的界面再补 Agent 接口。应该从架构设计层面就把两类用户分开优化：

- **共享层**：模型路由、项目数据结构、资产管理、工作流引擎
- **人类入口**：可视化画布/编辑器，优化空间记忆和直觉操作
- **Agent 入口**：Skill 接口，优化意图表达和状态管理

### 10. Skill 中间层 — 不是 API 也不是一键出片

LibTV 的 Skill 设计解决了一个关键问题：Agent 调用能力时，抽象粒度应该在哪一层？

| 层级 | 特点 | 问题 |
|------|------|------|
| 碎片化 API | 一次调一个功能 | Agent 需要来回调几十次 |
| 一键出片 | 黑盒全自动 | 不同任务没法差异化 |
| **Skill（中间层）** | **意图到编排** | **Agent 表达意图，系统完成编排** |

Skill 不是把 GUI 操作翻译成代码调用（那只是 API），而是让 Agent 用自然语言表达意图，由系统侧完成编排和决策。每个 Skill 里有完整的决策链路，但 Skill 之间可自由组合。

**对自建系统的启示：**

自建系统的对外接口应该设计为三层：

```
底层 API    → 单个模型调用、单个操作（内部用）
Skill 层    → "生成一部漫剧"、"复刻这个视频风格"等意图级接口（Agent 用）
GUI 层      → 画布/编辑器（人类用）
```

Skill 层是关键创新点。它遵循 OpenClaw Skill 规范，可以被主流 Agent（OpenClaw、KimiClaw 等）直接调用。自建系统如果从一开始就设计成 Skill 接口，就能天然融入 Agent 生态。

### 11. Agent 状态管理 — sessionId / projectUuid / projectUrl

Agent 没有人类的"空间记忆"（人类看一眼画布就知道整体状态），需要系统显式管理状态。LibTV 的方案是每次 Agent 创作返回三要素：

```
sessionId    → 对话级状态：Agent 持续查询创作进展
projectUuid  → 项目级状态：角色约束、前面镜头的色调、已完成的分镜
projectUrl   → 人机切换桥梁：人类随时打开接管编辑
```

**对自建系统的启示：**

自建系统的 Agent 接口需要设计类似的状态管理：

```python
# Agent 发起创作请求
response = create_drama(prompt="生成一部西游题材的漫剧，10集")

# 返回三要素
response.session_id     # Agent 后续查询/继续操作用
response.project_id     # 项目持久化，跨 session 可恢复
response.editor_url     # 人类可随时在浏览器中接管
```

核心原则：**Agent 交付的不是 mp4 死文件，而是可继续编辑的项目**。这让人机协作成为可能——Agent 做 80% 的工作，人类做最后 20% 的审美判断和精修。

### 12. 工作流模板 = 可流通的审美资产

LibTV 的工作流模板不只是"技术参数模板"，还承载了创作者的审美判断：镜头时长偏好、构图倾向、色调范围、节奏结构。这些模板可以在社区流通，Agent 调用时也会带着创作者的审美印记。

**对自建系统的启示：**

前面从火山剧创提炼的"分镜策略库"（第 5 节）可以进一步升级为**可流通的工作流模板**：

```
策略库（火山剧创启发）      → 规则化的镜头策略，结构化标签匹配
工作流模板（LibTV 启发）    → 带审美判断的完整创作流程，可复用可流通
```

两者结合：
- 策略库解决"怎么拍"的规则问题（镜头公式）
- 工作流模板解决"拍出什么感觉"的审美问题（参数偏好）
- 社区流通机制让好的模板能被发现和复用

### 13. Agent 创作的经济学 — 批量生成需要低单价

LibTV 的定价策略（比 Runway/Pika 低 76-92%）不只是商业竞争，而是**Agent 生态的基础设施需求**。

Agent 没有人类的直觉审美，它的办法是靠量：每个镜头生成好几个版本，按一致性、构图、风格匹配度自动筛选。用算力换审美。这意味着 Agent 的调用频次是人类的 3-10 倍。

**对自建系统的成本模型启示：**

```
人类创作成本 = 单次生成成本 × 平均重试次数（3-5次）
Agent 创作成本 = 单次生成成本 × 批量生成数（10-30次）× 筛选率

→ Agent 单价必须是人类单价的 1/3 到 1/10
→ 模型路由不只选"最好的"，还要选"性价比最高的"
→ 质量评估 Agent（第 7 节）需要尽早介入，减少无效生成
```

结合巨日禄的成本控制经验（第 8 节），自建系统的成本架构应该是：

1. **模型路由**按场景选模型（巨日禄经验）
2. **Skill 层**根据意图决定用"效率模式"还是"质量模式"（巨日禄双工作流经验）
3. **批量生成 + 自动比选**由质量评估 Agent 筛选（LibTV 经验）
4. **成本跟踪**精确到每个镜头的实际花费（巨日禄经验）

## 来自小云雀短剧 Agent 调研的架构启发

> 小云雀是字节跳动（剪映团队）推出的 C 端短剧 Agent，搭载 Seedance 2.0。它的**故事理解能力、全局角色生命周期管理、原生音画一体、并行化生成**对自建系统有重要参考价值。详细调研见 [tools/xiaoyunque.md](../tools/xiaoyunque.md)。

### 14. 故事理解优先 — Agent 层的核心能力

小云雀最大的突破不在画质（那是 Seedance 2.0 的功劳），而在 Agent 层的**故事理解**能力。它能分析角色在不同情境下的行为一致性、理解叙事弧的情感变化、梳理世界观和人物关系。

**对自建系统的启示：**

Agent 层应该分两个阶段：

```
阶段一：故事理解（串行，不可跳过）
  剧本 → 世界观解析 → 人物关系图 → 每场戏的情感基调 → 角色行为逻辑
  输出："故事蓝图"（story blueprint）

阶段二：生产执行（可并行）
  故事蓝图 → 角色资产生成 → 分镜生成 → 视频生成 → 配音配乐
```

故事蓝图是整个 Agent 系统的"上帝视角"——后续所有环节都基于这个蓝图做决策。这比"逐集逐镜头处理"的方式在一致性上好得多。

### 15. 全局角色生命周期管理 — 三种一致性方案的综合

目前调研到的角色一致性方案有三种，各有优劣：

| 方案 | 代表工具 | 层面 | 优势 | 劣势 |
|------|---------|------|------|------|
| 融图 | 巨日禄 | 图像合成 | 视觉层面像素级控制 | 每个镜头需手动调 |
| 资产继承 | FreeToon | 语义追踪 | 适合长篇角色状态变化 | 依赖初始资产准确性 |
| 全局角色管理 | 小云雀 | 语义+视觉 | 理解身份连续性和行为逻辑 | 偶有偏差仍需人工修 |

**自建系统应综合三者：**

```
全局角色管理（小云雀启发）
  → 剧本解析时构建角色全生命周期档案
  → 分析称谓变化、年龄变化、服饰变化等

资产继承（FreeToon 启发）
  → 为角色每个状态维护视觉资产
  → 追踪"第3集穿红衣"→"第7集换盔甲"的状态变化

融图注入（巨日禄启发）
  → 每次生成时注入对应状态的参考图
  → 保证视觉层面的像素级一致性
```

### 16. 原生音画一体 — 配音应在生成阶段注入

小云雀的"声音从画面里长出来"的方案，比"先生成视频再贴配音"在一致性上更优，特别是口型同步效果。

**对自建系统的启示：**

传统流水线：
```
剧本 → 画面生成 → 视频生成 → TTS 配音 → 口型后处理 → 合成
```

应探索的方案：
```
剧本 → 画面+语音联合生成（Seedance 2.0 已支持） → 原生口型同步
```

当前 Seedance 2.0 已原生支持口型同步，说明底层模型技术已具备。自建系统在模型路由层应优先选择支持音画联合生成的模型，将口型同步从"后处理"变成"生成约束"。

### 17. 并行化非线性生成 — 工作流引擎设计

小云雀在完成剧本解析后，为整部剧构建"数字资产档案"和"创作蓝图"，然后可以并行生成多集多镜头。

**对自建系统的工作流引擎设计：**

```
串行阶段（必须按序完成）：
  剧本解析 → 故事蓝图 → 角色资产生成 → 分镜策略匹配

并行阶段（基于蓝图，可多线程执行）：
  ├── 第1集镜头1-10 并行生成
  ├── 第1集镜头11-20 并行生成
  ├── 第2集镜头1-10 并行生成
  └── ...

约束传递：
  同一角色的关键镜头（首次出场、重大变化）应优先生成
  后续镜头基于这些"锚点镜头"的视觉参考生成
  → 既保证一致性，又最大化并行度
```

### 18. 免费 C 端工具的竞争压力 — 自建系统的差异化方向

小云雀免费 + Seedance 2.0 顶级画质 + 字节分发闭环，对整个行业形成降维打击。自建系统必须在以下维度建立差异化：

| 维度 | 小云雀的限制 | 自建系统的机会 |
|------|------------|--------------|
| 模型路由 | 绑定 Seedance 2.0 | 30+ 模型自由选择（参考 LibTV） |
| 可编程性 | 封闭平台 | API + Skill 接口，可集成到自动化流水线 |
| 分发平台 | 绑定字节生态 | 不绑定特定平台，支持多平台分发 |
| 策略库 | 无内置策略 | 可定制爆款策略库（参考火山剧创） |
| 质量评估 | 人工判断 | Agent 自动评估 + 重试（所有工具都未实现） |
| 数据所有权 | 数据在字节云上 | 可私有部署 |

## 视频拼接与合成 — 从短片段到完整剧集的自动化方案

> 当前 AI 视频生成模型（Seedance 2.0、Kling、Vidu 等）单次只能生成 5-15 秒片段。一集 1-2 分钟的剧集需要拼接 8-20+ 个片段，一部 60 集的剧需要组装数千个片段。**拼接不是后处理步骤，是规模化生产的核心瓶颈。**

### 19. 现有工具的拼接方案对比

调研的三个开源工具和主要商业工具，采用了截然不同的方案：

| 工具 | 方案 | 转场效果 | 可自动化 | 可扩展性 |
|------|------|---------|---------|---------|
| 火宝短剧 | FFmpeg 直接拼接（fluent-ffmpeg） | 无（硬切） | ✅ 全自动 | 低 |
| waoowaoo | Remotion 可编程视频（React 组件） | 3 种基础转场 | ⚠️ 仅浏览器预览，服务端渲染未实现 | 高（代码驱动） |
| Jellyfish | 内置时间线编辑器 | 未知 | ❌ 需手动拖拽 | 中 |
| 火山剧创 | 导出到剪映/CapCut | 完整（依赖编辑器） | ❌ 需人工操作 | N/A |
| 小云雀 | Agent 自动组装 + 音画一体 | 未公开 | ✅ 全自动 | 低（闭源） |

**核心结论：没有一个开源工具优雅地解决了自动化拼接问题。**

### 20. 自动化拼接的四层方案

从底层到上层，业界有四种可用于自动化的技术路线：

#### 方案 A：FFmpeg `xfade` 滤镜 — 性能最优的自动化底座

FFmpeg 原生支持 [44 种转场效果](https://ottverse.com/crossfade-between-videos-ffmpeg-xfade-filter/)（dissolve、fade、wipe、slide、radial、pixelize 等），通过 `xfade` 滤镜实现：

```bash
# 两个片段 + dissolve 转场
ffmpeg -i clip1.mp4 -i clip2.mp4 \
  -filter_complex "xfade=transition=dissolve:duration=0.5:offset=4.5" \
  output.mp4
```

多片段拼接时需**链式串联** `xfade`，每步输出作为下一步输入，offset 累计计算（总时长 - 转场重叠）。已有现成自动化脚本：

- [Python 自动 offset 计算脚本](https://gist.github.com/royshil/369e175960718b5a03e40f279b131788) — 处理不同分辨率/帧率输入
- [Node.js `ffmpeg-transitions` 库](https://github.com/Rickkorsten/ffmpeg-transitions) — 支持每对片段不同转场
- [xfade-easing](https://github.com/scriptituk/xfade-easing) — 扩展 CSS 缓动曲线和 GLSL 自定义转场

**注意**：xfade 要求所有输入的 timebase 和帧率一致，需先用 `settb=AVTB,fps=30/1` 归一化。

**适合**：后端自动化管线。性能最好，Go/Python 均可通过 `exec.Command` / `subprocess` 调用。缺点是复杂合成（多轨、字幕叠加、BGM 混音）需要手写 `filter_complex`，可读性差。

#### 方案 B：Remotion Lambda — 无服务器分布式渲染

[Remotion Lambda](https://www.remotion.dev/docs/lambda) 将 React 组件描述的视频合成部署到 AWS Lambda 上渲染：

```
1. Remotion React 项目部署到 S3
2. renderMediaOnLambda() 触发主 Lambda
3. 主 Lambda 按帧数分发多个 renderer Lambda 并行渲染
4. 各 renderer 通过 Response Streaming 回传二进制 chunk
5. 主 Lambda 拼接 chunk → 上传最终 MP4 到 S3
```

可通过 [API Gateway + SQS](https://www.remotion.dev/docs/lambda/sqs) 构建全自动队列管线。大多数用户渲染几分钟视频只花几美分。

**适合**：已用 React/TypeScript 写合成逻辑的项目。优势是转场效果用 React 组件编程，扩展性最好。劣势是需要 AWS 基础设施，且引入了 Node.js/React 技术栈。

#### 方案 C：MoviePy — Python 生态的快速方案

[MoviePy](https://github.com/Zulko/moviepy)（v2.2.1）底层也是 FFmpeg，但提供 Python API，和 AI 管线天然集成：

```python
from moviepy import VideoFileClip, concatenate_videoclips

clips = [VideoFileClip(f"shot_{i}.mp4") for i in range(10)]
for i in range(1, len(clips)):
    clips[i] = clips[i].crossfadein(0.5)
final = concatenate_videoclips(clips, method="compose")
final.write_videofile("episode.mp4")
```

**适合**：Python 技术栈的快速原型。劣势是比直接 FFmpeg 慢（Python 数据搬运开销），大量片段拼接时有已知内存问题。

#### 方案 D：Agent 驱动的 Assembly Pipeline

[ViMax](https://github.com/HKUDS/ViMax)（港大，2.6k stars，HKUDS 实验室）采用 Multi-Agent 架构，其中 **Assembly Agent** 专门负责拼接：

```
Script Agent → Storyboard Agent → Visual Agent → Quality Agent → Assembly Agent → Coordinator Agent
```

Assembly Agent 自动决策拼接顺序、转场类型和时间节奏，底层仍用 MoviePy/FFmpeg，但决策逻辑由 Agent 驱动。

**适合**：想做端到端自动化、Agent 驱动全流程的架构。值得参考其 Assembly Agent 的决策逻辑设计。

### 21. 模型层面的长视频生成 — 未来可能消除拼接需求

[Stable Video Infinity](https://github.com/vita-epfl/Stable-Video-Infinity)（ICLR 2026 Oral，EPFL）尝试从根本上绕过拼接问题——让模型**原生生成任意长度视频**。

核心方法：**Error-Recycling Fine-Tuning** — 把模型自身生成的错误回收为训练信号，让模型学会自我纠错，从而在自回归逐段生成时保持时序一致性。支持多场景转场、250 秒以上连续视频。

**当前限制**：仍是研究项目，模型资源消耗大，不适合直接用于批量生产。但代表了长期方向——拼接问题最终可能被模型能力本身解决。

### 22. 对自建系统的拼接层架构建议

结合调研结论，自建系统的拼接层应按阶段演进：

| 阶段 | 推荐方案 | 理由 |
|------|---------|------|
| MVP | FFmpeg `xfade` 命令生成器 | Go 调 FFmpeg 最直接，44 种转场够用，性能最好 |
| 进阶 | Assembly Agent（参考 ViMax） | Agent 自动决策转场类型和节奏，不再硬编码 |
| 长期 | 关注 SVI 类模型 | 当模型能直接生成 2+ 分钟视频时，拼接层可能退化 |

**MVP 阶段的具体实现思路：**

```
输入：有序的 []Shot，每个 Shot 有 videoPath、durationSec、importance
输出：一条自动计算 offset 的 FFmpeg 命令

实现：
1. 所有输入先归一化帧率和分辨率（settb + fps + scale）
2. 根据 Shot.importance 自动选转场类型：
   - S 级镜头（高潮/转折）：用 dissolve 1.0s
   - A 级镜头（叙事推进）：用 fade 0.5s
   - B/C 级镜头（过渡/填充）：用 fade 0.3s 或硬切
3. 链式生成 xfade filter_complex，自动累计 offset
4. 音频轨单独处理：acrossfade 做音频交叉淡入淡出
5. 字幕轨用 ASS 格式烧录或 MKV 软字幕
6. 一次 exec.Command("ffmpeg", args...) 完成
```

**关键设计原则：**
- 转场决策应由 Agent 根据叙事结构做出（高潮用慢溶解，日常用快切），不应硬编码
- 拼接层是纯执行层，不做创意决策——创意决策在分镜阶段已完成
- 需要归一化所有输入片段的帧率/分辨率，否则 xfade 会失败
- 音频和视频应分轨处理后合并，避免 filter_complex 过于复杂

## 视频复刻 — 用多模态模型倒推提示词

> 来源：[@binghe 2026-04-03](https://x.com/binghe/status/2040096430929686600)，247 赞 / 303 收藏

### 23. 视频理解 → 倒推提示词 → 复刻爆款

核心思路：用多模态模型（视频理解能力）分析一段爆款视频，**倒推出完整的生成提示词**，然后用视频生成模型复刻。

**工作流：**

```
爆款视频 → 多模态模型分析（视频理解）→ 结构化提示词 → 视频生成模型 → 复刻版本
```

**可分析的维度：**
- 画面内容：场景、角色、动作、构图
- 运镜语言：推拉摇移、景别切换、节奏
- 内容钩子：开头设计、反转节点、悬念结构
- BGM 卡点：音乐节奏与画面的配合关系
- 产品展示逻辑（电商场景）

**模型选择：**
- Qwen3.5-omni：视频理解能力强，成本极低（百万 Token 几毛钱），同步分析（非逐帧分析后汇总）
- Gemini：逐帧分析后汇总，成本和效率相对更高

**应用场景：**
- 短剧制作：分析爆款短剧的镜头语言、节奏和钩子设计，复刻其"爆款因子"
- 电商带货：拆解爆款带货视频，复刻内容结构（也是短剧的变现场景之一）
- 抖音"暗水印"应对：平台对搬运素材打暗水印导致投流失效，复刻而非搬运可绕过

**对 short-maker 的价值：**

这和 [script-sourcing.md](script-sourcing.md) 中"路径 4：提炼情绪公式"以及"路径 2：爆款换皮"直接相关。视频理解模型可以自动化提炼过程：

```
以前（人工）：人看爆款视频 → 手动总结镜头语言和节奏 → 手动写提示词
现在（自动化）：爆款视频 → Qwen3.5-omni 分析 → 结构化提示词 → Pipeline 生成
```

可以集成到 Pipeline 的前置环节：给一个参考视频，自动提取其视觉风格、运镜策略、节奏结构，作为生成参数注入到分镜和视频生成阶段。
