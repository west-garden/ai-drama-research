# Jellyfish — AI 短剧工厂

> 开源的一站式 AI 短剧生产工具，从剧本到成片全流程闭环

- **官网/仓库:** https://github.com/Forget-C/Jellyfish
- **类型:** 端到端短剧生产平台（开源）
- **协议:** Apache-2.0
- **技术栈:** React 18 + TypeScript（前端）/ FastAPI + Python（后端）
- **Stars:** 2,250（截至调研日期，创建不到一个月）
- **创建时间:** 2026-03-06
- **调研日期:** 2026-04-03

## 核心能力

Jellyfish 是一个**开源的 AI 短剧生产工具**，目标是覆盖短剧制作的完整链条：

**剧本输入 → 智能分镜 → 角色/场景/道具一致性管理 → AI 视频生成 → 后期剪辑 → 一键导出成片**

核心要解决的问题：
1. **AI 生成的一致性** — 人物/场景漂移是 AI 视频生产最大的痛点
2. **工业化流程** — 让短剧制作从"手工抽卡"变成"流水线生产"
3. **资产复用** — 角色、场景、道具、服装不是一次性的，要能跨章节复用

## 工作流

```
剧本文本
  ↓
智能精简（script_simplifier_agent）
  ↓
分段（script_divider_agent）
  ↓
元素提取（element_extractor_agent）
  ├─→ 角色画像分析（character_portrait_analysis_agent）
  ├─→ 场景信息分析（scene_info_analysis_agent）
  ├─→ 服装分析（costume_info_analysis_agent）
  └─→ 道具分析（prop_info_analysis_agent）
  ↓
实体合并 + 一致性检查（entity_merger + consistency_checker）
  ↓
分镜提示词生成（shot_frame_prompt_agents）
  ↓
AI 图片/视频生成（对接外部模型 API）
  ↓
时间线剪辑 → 导出成片
```

## 技术架构

### 前端

- React 18 + TypeScript + Vite
- Ant Design + Tailwind CSS
- React Flow（Agent 工作流可视化编辑）
- OpenAPI 自动生成请求代码（`front/src/services/generated/`）
- 端口：7788

### 后端

- Python / FastAPI
- SQLAlchemy + MySQL（开发可用 SQLite）
- RustFS（S3 兼容对象存储，存生成素材）
- LangChain/LangGraph Agent 链
- 端口：8000（`/docs` 为 Swagger）

### Agent 系统（核心亮点）

后端实现了完整的多 Agent 链用于剧本处理，位于 `backend/app/chains/agents/`：

| Agent | 职责 |
|-------|------|
| `script_simplifier_agent` | 剧本精简 |
| `script_divider_agent` | 剧本分段 |
| `element_extractor_agent` | 元素提取 |
| `character_portrait_analysis_agent` | 角色画像分析 |
| `scene_info_analysis_agent` | 场景信息分析 |
| `costume_info_analysis_agent` | 服装信息分析 |
| `prop_info_analysis_agent` | 道具信息分析 |
| `consistency_checker_agent` | 一致性检查 |
| `entity_merger_agent` | 实体合并去重 |
| `variant_analyzer_agent` | 变体分析 |
| `shot_frame_prompt_agents` | 分镜提示词生成 |
| `output_compiler_agent` | 输出编译 |

这些 Agent 通过 LangGraph（`backend/app/chains/graphs.py`）编排成 DAG 工作流。

### 模型对接

支持多供应商模型管理：
- 文本模型：OpenAI / Claude / 通义 / 混元 等
- 图片模型：多种（具体未列出）
- 视频模型：Runway / Kling / Luma 等
- 每类可设默认模型，支持快速测试连接

### 部署

Docker Compose 一键启动（MySQL + RustFS + Backend + Front），也支持前后端分离开发。

## AI 能力边界

- **能自动完成的：** 剧本精简、分镜提取、角色/场景/道具提取与分析、分镜提示词生成、一致性检查
- **需要人工介入的：** 分镜微调（景别/角度/运镜等精细控制）、资产确认与关联、生成结果筛选、时间线剪辑
- **完全做不到的：** 剧本创作、音效/配乐的实际生成（仅生成提示词）

## 可编程性

- **API:** 完整的 RESTful API（FastAPI + OpenAPI 文档）
- **批量操作:** 支持
- **可集成到自动化流水线:** 高。开源 + 标准 API + Docker 部署，可二次开发
- **Agent 可扩展:** 工作流编辑器支持自定义 Agent 节点编排

## 优点

- **开源 + Apache-2.0** — 可自由修改和商用，不绑定任何平台
- **多 Agent 架构** — 不是单一 LLM 调用，而是 12 个 Agent 组成的流水线，职责清晰
- **一致性方案完整** — 全局种子 + 风格继承 + 资产库双层体系 + 一致性检查 Agent
- **工作流可编辑** — 类 Dify 的节点式编辑器，Agent 工作流可定制
- **多模型路由** — 不绑定单一模型供应商，灵活切换
- **技术栈现代** — React 18 + FastAPI + LangGraph，代码质量和架构设计清晰
- **增长势头强** — 开源不到一个月 2250 stars，市场需求明确

## 缺点

- **项目太新** — 2026.03.06 创建，核心流程未稳定，不适合直接生产使用
- **不做剧本生成** — 和其他工具一样，需要外部提供剧本
- **语音/配音缺失** — 没有语音合成、配音相关能力
- **无 SaaS 版本** — 纯自部署，对非技术用户门槛高
- **文档尚不完善** — 虽然有 Hugo 文档站，但内容还在建设中
- **单人维护风险** — 目前看是个人/小团队项目，长期维护存在不确定性

### 核心差异点

#### 1. 架构哲学完全不同

- **Jellyfish** — Agent-first。12 个 LangGraph Agent 组成 DAG 工作流，每个 Agent 职责单一。Agent 工作流还可以通过可视化编辑器自定义。**设计重点在 Agent 编排层。**
- **waoowaoo** — 基础设施-first。自建工作流运行时（租约+事件溯源+心跳+看门狗）、完善的计费系统（冻结-结算-退款）、BullMQ 高并发队列。**设计重点在可靠性和分布式执行。**
- **火宝短剧** — 产品-first。Skill-as-prompt 让非技术用户可在 UI 上编辑 Agent 行为，零依赖部署，一键预设快速上手。**设计重点在用户体验和低门槛。**

#### 2. Agent 系统的深度差异

| 维度 | Jellyfish (12 Agent) | waoowaoo (无 Agent) | 火宝 (5 Agent) |
|------|---------------------|-------------------|---------------|
| Agent 数量 | 12 个专用 Agent | 0（4 阶段流水线函数） | 5 个专用 Agent |
| 编排方式 | LangGraph DAG | 自建 run-runtime + workflow-engine | Mastra 框架 |
| 可视化编辑 | ✅ React Flow 工作流编辑器 | ❌ | ❌ |
| Prompt 管理 | Agent 内置 | Prompt Catalog（但核心路径不用） | SKILL.md（可 UI 编辑） |
| 一致性检查 | ✅ 独立 Agent | ❌ | ❌ |
| 实体合并去重 | ✅ 独立 Agent | ❌ | ✅ 提取时去重 |

Jellyfish 的 Agent 系统是三者中最精细的——有独立的一致性检查 Agent、实体合并 Agent、变体分析 Agent，这些在另外两个项目中完全没有。

#### 3. 分镜生成的设计差异

| 维度 | Jellyfish | waoowaoo | 火宝 |
|------|-----------|----------|------|
| 方式 | Agent 链（精简→分段→提取→提示词） | 4 阶段流水线（规划→摄影→表演→细节） | 单步 Agent（17 字段一次产出） |
| 分镜 prompt 生成 | 独立 Agent | Phase 3 细节充实 | 分镜拆解时同步生成 |
| 校验机制 | 一致性检查 Agent | 无（Prompt Catalog 被绕过） | 无 |
| 人工干预点 | 每步可编辑 | 每步可编辑 | 单步结果编辑 |

waoowaoo 的 4 阶段方案在工程上最合理（渐进细化），Jellyfish 的 Agent 链在语义理解上最深（有一致性校验），火宝的单步方案最简单但也最脆弱。

## 对自建系统的启发

### 1. Agent 链设计可直接参考

Jellyfish 的 12 个 Agent 分工是目前能看到的**最详细的开源实现**。从剧本精简到一致性检查，每个环节都有独立 Agent，通过 LangGraph 编排。这个架构设计可以直接作为自建系统的参考蓝图。

### 2. 资产管理的双层体系

"项目级资产库 vs 全局资产库"的设计值得借鉴。项目级资产是当前短剧专用的，全局资产是跨项目复用的（比如同一个 IP 的角色可以出现在不同短剧中）。

### 3. 一致性检查作为独立 Agent

不只是在生成时靠种子/风格保一致性，还有专门的 `consistency_checker_agent` 在出片前做校验。这是一个"质量门禁"的设计思路。

### 4. 开源代码可以直接学习

作为 Apache-2.0 项目，可以直接阅读其 Agent prompt 设计、LangGraph 工作流编排、FastAPI 后端架构等具体实现。特别是 `backend/app/chains/` 目录下的 Agent 代码，是实打实的参考材料。

### 5. 工作流可视化编辑是加分项

用 React Flow 实现的 Agent 工作流编辑器，让非技术用户也能调整 Agent 流程。如果自建系统面向的不只是开发者，这个能力很有价值。
