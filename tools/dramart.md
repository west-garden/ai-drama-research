## 产品定位

火山剧创是火山引擎推出的**工业级短剧生产 Agent**，核心卖点是：
- 接入字节自家的 **Seedance 2.0** 视频生成模型
- 采用**多 Agent 协同架构**（不是单一工具，是多个 Agent 分工合作）
- 制作周期缩短 **80%** 以上
- 面向 B 端承制方和制作团队

## 核心能力与特色

### 1. 剧本直出分镜 — "懂行"的 AI 导演

用户上传剧本后，Agent **直接读取剧情逻辑和叙事节奏**，拆解出符合影视工业标准的专业分镜脚本。不需要手动将剧本转化为提示词。

> 对比巨日禄：巨日禄也支持剧本上传自动分镜，但火山剧创强调的是"深度理解"剧情逻辑，而非简单的文本分割。

### 2. 全局风格锁定 + 素材资产库

- 支持 **2D / 3D / 仿真人** 多种视觉风格
- 创作初期可自定义**角色、变装、场景环境、关键道具**
- 形成**可复用的素材资产库**
- Agent 能追踪上下文状态，大幅减少人设崩坏和场景穿帮

> 对比巨日禄：巨日禄通过"融图"技术保持一致性，火山剧创通过"全局风格锁定 + 上下文追踪"实现。思路不同，目标一样。

### 3. 内置 200+ 爆款镜头策略

平台内置短剧行业的**爆款镜头策略库**（冲突、反转、高潮三大核心节点），并引入**多 Agent 校验机制**，自动识别分镜中的人物/场景/道具是否配置完整。

> 这是火山剧创最大的差异化：把短剧行业的"爆款方法论"内置到了 Agent 中，不依赖用户的个人经验。

### 4. Seedance 2.0 集成

接入字节自家的 Seedance 2.0 模型：
- 分镜信息解析更精准
- 提示词描述更贴合导演真实创作需求
- 镜头直接可用率大幅提升（降低抽卡率）

### 5. 一键导出剪映

确认的分镜序列可一键导出至剪映（CapCut），与字节系剪辑工具无缝衔接。

### 6. 企业级协同

- 企业级资产管理
- 多人协同创作
- 历史版本回溯

## 技术架构亮点

### 多 Agent 协同设计

火山剧创不是单一 Agent，而是**多个 Agent 分工协作**的工业流水线：

```
剧本解析 Agent → 素材设定 Agent → 分镜解析 Agent → 视频生成 Agent → 剪辑导出 Agent
                                        ↑
                                  多 Agent 校验机制
                                 （检查人物/场景/道具完整性）
```

这种架构与巨日禄的"单一平台+人工调度"完全不同，更接近我们想做的 Agent 驱动系统。

### 背靠字节生态

- **Seedance 2.0** — 字节自研视频生成模型
- **剪映/CapCut** — 无缝导出到字节系剪辑工具
- **巨量引擎** — 字节的广告投放系统（漫剧变现的核心渠道）
- **红果短剧** — 字节的短剧分发平台

整个链条是：**火山剧创生产 → 剪映后期 → 红果/抖音分发 → 巨量引擎投放**。字节在试图闭环。

## 优点

- **多 Agent 架构** — 架构设计更先进，接近工业化流水线思路
- **内置爆款策略** — 将行业 know-how 编码进系统，降低对用户经验的依赖
- **字节生态闭环** — 从生产到分发的完整链条
- **Seedance 2.0** — 直接用自家最好的模型，版本跟进零延迟
- **企业级协同** — 一开始就考虑了团队协作和资产管理

## 缺点

- **刚邀测** — 2026.03.30 才发布，没有经过市场验证
- **绑定字节生态** — 模型只用 Seedance，缺乏巨日禄的多模型路由灵活性
- **企业门槛** — 评论区用户反馈"个人玩不了，只能企业才能玩"
- **不做剧本生成** — 和巨日禄一样，需要外部提供剧本
- **配音/口型/字幕** — 公开信息未提及，可能仍需外部工具

## 对自建系统的启发

### 1. 多 Agent 协同是正确的架构方向

火山剧创验证了"多 Agent 分工协作"这个架构思路。每个 Agent 负责一个环节（剧本解析、素材设定、分镜、视频、剪辑），通过校验机制保证质量。**这和我们想做的 Agent 驱动系统高度一致。**

### 2. 爆款策略库是差异化利器

内置 200+ 爆款镜头策略，把行业经验变成可编程的规则。自建系统应该考虑类似的"策略库"设计——把短剧的叙事模式、镜头语言、节奏控制编码成 Agent 可调用的知识。

### 3. 上下文追踪解决一致性

不用"融图"这种图像处理手段，而是让 Agent 在语义层面追踪上下文状态（角色状态、场景变化、道具位置）。这是一种更"Agent-native"的方案，值得借鉴。

### 4. 字节的威胁和机会

字节在做全链路闭环（生产→剪辑→分发→投放），但也意味着**不在字节生态内的创作者需要替代方案**。一个开放的、不绑定特定平台的 Agent 系统，反而有差异化空间。

### 5. 单一模型 vs 模型路由

火山剧创只用 Seedance 2.0，巨日禄用多模型路由。杰夫说过"没有哪个模型能断层领先超过两个月"，如果 Seedance 被其他模型超越，火山剧创的灵活性就成问题。**自建系统应该走模型路由的路线。**
