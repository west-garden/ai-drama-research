# waoowaoo (AI Video Studio)

> 开源全流程 AI 影视生产平台，从小说文本到短剧视频的端到端自动化

- **官网:** https://github.com/saturndec/waoowaoo
- **类型:** 端到端平台（开源）
- **Stars:** 10,778 | **Forks:** 2,399
- **技术栈:** TypeScript / Next.js 15 / React 19 / Prisma + MySQL / BullMQ + Redis / Remotion
- **调研日期:** 2026-04-03

## 核心能力

将小说/故事文本作为输入，通过 AI 全自动生成短剧视频。覆盖剧本解析、角色设计、场景生成、分镜规划、图片生成、视频生成、配音合成、口型同步、视频剪辑的完整链路。用户自带 API Key，系统充当编排层。

## AI 能力边界

- **能自动完成的：** 小说解析→剧本→分镜→媒体生成的完整链路；角色/场景的 AI 创建与变体生成；多角色配音分配与合成
- **需要人工介入的：** 角色形象确认与选择；分镜结果审核与调整；视频剪辑的精细调整；API Key 配置
- **完全做不到的：** 实时预览渲染；复杂视频特效（仅基础转场）；AI 配乐生成；多角色互动的连贯视频

## 架构评价

### 整体架构设计：7.5/10

采用 **Next.js 单体应用 + 独立 Worker 进程** 的架构：

```
[浏览器] → [Next.js Server + API Routes] → [MySQL via Prisma]
                     ↓
              [BullMQ 队列] ↔ [Redis]
                     ↓
              [4 类 Worker 进程]
                     ↓
              [外部 AI API] + [MinIO 存储]
```

加上 Watchdog 看门狗进程和 Bull Board 管理面板。

**评价：** 对于一个开源项目来说，这个架构选择是务实的。Next.js 全栈减少了部署复杂度，BullMQ 提供了可靠的异步任务处理。但单体架构意味着所有组件共享一个进程的资源上限，规模化时会遇到瓶颈。

### 各模块设计深度评价

#### 工作流引擎 (run-runtime)：8/10 — 过度设计但质量高

两层架构：底层 `run-runtime/` 是通用的数据库支撑的 DAG 执行运行时，带租约机制（lease-based distributed ownership）和事件溯源（event sourcing）；上层 `workflow-engine/` 提供声明式步骤定义和依赖图。

**亮点：**
- 租约系统（claim/renew/release）+ 心跳存活检测设计正确
- 事件溯源通过原子性的序列号递增 + 投影更新，防止事件乱序
- 重试时感知 DAG 依赖，重试 `analyze_characters` 会自动失效下游所有步骤
- `reconcile.ts` 做跨实体一致性修复——检测到 Task 已终止但 Run 仍在运行的不一致状态

**问题：**
- `service.ts` 1200 行，混杂了 CRUD、事件投影、步骤管理、检查点管理等多重职责
- **实际上没有 DAG 调度器**——注册表定义了步骤依赖，但没有拓扑排序执行器。"DAG 引擎"其实只是状态追踪器，实际调度发生在 Worker 代码中
- 目前只有 2 种工作流（story-to-script 和 script-to-storyboard），泛化引擎的复杂度相对于使用场景偏重

#### 计费系统 (billing)：9/10 — 全项目最成熟的模块

冻结-结算-退款（freeze-settle-refund）模式：任务开始前冻结预估费用，成功后结算（多冻少退），失败后回滚。

**亮点：**
- 幂等性是一等公民，`freezeBalance` 支持 `idempotencyKey`，重复提交不会双重扣费
- 乐观并发控制——`updateMany` 带 `balance >= requiredAmount` 条件，无需显式锁
- 金额处理用 `Decimal(18,6)` + `MONEY_SCALE = 6` + epsilon 比较，防浮点漂移
- 支持工作流中途扩展冻结额度（`increasePendingFreezeAmount`）

**问题：**
- 待处理冻结没有 TTL 过期清理，泄漏的冻结会永久减少可用余额
- `rollbackFreeze` 失败时只 log 不抛异常，调用方不检查返回值就会静默丢钱

#### Worker 系统 (BullMQ)：8/10 — 可靠的中间件模式

`withTaskLifecycle` 统一封装了心跳、状态转换、计费结算/回滚、事件发布、重试决策。

**亮点：**
- 每个 Worker 自动获得一致的错误处理和计费逻辑
- `withUserConcurrencyGate` 实现用户级并发限制，防止单用户垄断 Worker
- 可重试/不可恢复错误的区分正确

**问题：**
- `shared.ts` 730 行，混合了太多关注点
- 双发布策略（legacy Task 事件 + 新 GraphRun 事件）增加维护负担

#### 分镜生成 (storyboard-phases)：6/10 — 核心价值路径却是最粗糙的代码

4 阶段流水线是产品的核心差异化能力，但代码质量反而最差。

**问题：**
- **大量代码重复**——4 个阶段结构几乎相同（加载模板→构建上下文→调用 AI→解析 JSON→重试），只是参数不同，应该抽象为泛化函数
- **绕过了 prompt 校验系统**——`buildPrompt` 提供了变量校验和占位符匹配检测，但分镜生成直接用 `template.replace()` 手动替换，放弃了安全网
- **脆弱的 JSON 解析**——靠字符串查找 `[` 和 `]` 来提取 JSON，没有使用 LLM 的 structured output / JSON mode
- Phase 3 接收 `photographyRules` 参数但 `void` 掉了——未完成的重构残留
- 重试硬编码 2 次，无退避策略

#### 生成器抽象 (generators)：7/10 — 简洁但有漏洞

经典 Strategy + Factory 模式，添加新供应商只需实现接口 + 加 switch case。

**问题：**
- `BaseAudioGenerator` 没有重试逻辑，但图片和视频生成器有——不一致
- 没有异步轮询接口（`checkStatus`），异步任务结果的轮询在其他地方散落处理
- 重试退避是线性的（1s, 2s），对 API 限流应用指数退避更合适
- 10+ 供应商用 switch 分发，应该用 Map 注册表

#### Prompt 系统：7.5/10 — 设计精良但执行不彻底

`PROMPT_IDS` 编译时类型安全，catalog 声明必需变量，`buildPrompt` 运行时校验占位符与 catalog 是否一致。

**致命问题：** 核心的分镜生成代码不用它。安全网存在但最重要的路径绕过了它。

#### 视频编辑器 (Remotion)：5/10 — 早期阶段

237 行代码，基础转场效果（溶解、淡入淡出、滑动），两种字幕样式。没有错误边界、没有 memoization、没有复杂合成能力（图层、遮罩、文字特效、Ken Burns 等）。

#### 数据模型：7/10 — 索引完善但过度使用 JSON 列

30+ 模型，索引覆盖完善。`MediaObject` 的内容寻址存储设计（SHA256 + storageKey）很好。

**问题：**
- 大量结构化数据存为 `String @db.Text` JSON blob（角色列表、图片历史、分镜文本等），牺牲了可查询性和类型安全
- 项目资产和全局资产库存在严格的 schema 重复（Character/Location 各有两套结构相同的表）
- 没有软删除机制，全部级联硬删除

#### Watchdog：6.5/10 — 单实例设计

**问题：**
- 没有分布式锁，多副本部署时会重复处理任务
- 没有 SIGTERM 优雅关闭、没有健康检查端点
- 每轮最多处理 100 条任务，大积压时排空速度受限

### 设计风格总评

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构完整度 | 8/10 | 端到端链路完整，工作流引擎+队列+计费+监控一应俱全 |
| 代码质量 | 6.5/10 | 基础设施代码质量高（计费、租约），核心业务代码粗糙（分镜生成） |
| 抽象合理性 | 6/10 | 工作流引擎过度抽象（2 个用例撑不起通用引擎），分镜生成欠缺抽象（大段重复） |
| 可扩展性 | 7/10 | 多供应商策略好，但单体架构限制了水平扩展 |
| 可靠性设计 | 8/10 | 心跳+看门狗+租约+幂等计费，分布式系统经验丰富 |
| 安全性 | 7.5/10 | API Key 加密存储，日志脱敏，但无软删除和冻结过期清理 |
| 测试 | 7/10 | 测试框架完善（单元/集成/并发/契约/回归），30+ 守卫脚本 |

**一句话总评：** 基础设施工程能力远超业务代码质量——像是一个擅长后端系统设计的工程师做的产品，计费和分布式执行做得很好，但作为产品核心的分镜生成/prompt 工程反而是最粗糙的部分。

## 优点

- 功能链路完整度在开源项目中无出其右，从文本到视频的每一步都有覆盖
- AI 供应商集成极其全面（LLM 5 家、图片 6 家、视频 8 家、语音 3 家）
- 计费系统设计成熟，冻结-结算-退款模式在开源项目中罕见
- 分布式任务执行的可靠性设计（心跳、看门狗、租约、幂等）体现了工程经验
- 双语 prompt 系统（30+ 对中英模板）对国际化友好
- Docker Compose 一键部署，降低了上手门槛

## 缺点

- 核心业务代码（分镜生成）质量与基础设施代码质量严重不匹配
- 分镜生成绕过了自己构建的 prompt 校验系统，安全网形同虚设
- JSON 解析依赖字符串查找而非 LLM structured output，脆弱
- 单体架构，长期扩展受限
- 视频编辑器功能初级，只有基础转场
- 大量结构化数据存为 JSON blob，牺牲查询能力
- Watchdog 不支持多副本，分布式部署有风险
- 无开源协议

## 对自建系统的启发

### 值得借鉴

1. **冻结-结算-退款计费模式** — 任务开始前冻结预估费用，完成后结算差额，失败回滚。带幂等键和乐观并发。这是处理 AI API 费用的正确方式
2. **多阶段分镜流水线** — 将分镜生成拆为 4 个阶段（规划→摄影→表演→细节），每阶段职责明确，比一次性生成所有信息更可控
3. **心跳 + 看门狗 + 租约** 的组合确保了长时间 AI 任务（视频生成可能几分钟）的可靠执行
4. **多供应商策略 + 用户自带 Key** — 不绑定单一供应商，用户可以根据效果和成本自行选择
5. **MediaObject 中央注册表** — 用 SHA256 + storageKey 统一管理所有媒体文件，支持去重和迁移
6. **Prompt Catalog 的编译时类型安全** — 用 `as const` + 类型推导确保 prompt ID 和变量在编译时可检查

### 应该避免

1. **基础设施过度建设、业务代码欠缺投入** — 不要在通用引擎上花太多精力（只有 2 个工作流不需要通用 DAG 引擎），把精力花在核心价值路径上
2. **建了安全网但不用** — prompt 校验系统很好，但分镜生成不用它，等于没有。确保核心路径一定走安全路径
3. **JSON-in-column 模式** — 大量结构化数据存为文本 JSON blob 看似灵活，实际上牺牲了查询能力和类型安全，后期重构成本高
4. **线性重试而非指数退避** — 对外部 API 调用应该用指数退避 + 抖动
5. **单体架构包揽一切** — 前端、API、Worker 全在一个 Next.js 项目中，初期方便但长期受限
