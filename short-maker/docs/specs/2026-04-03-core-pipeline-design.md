# AI 短剧生产核心流水线设计

> 项目代号：Short Maker
> 日期：2026-04-03
> 状态：设计评审

## 一、产品定位

构建一个多 Agent 驱动的 AI 短剧生产工具，核心能力为"受限编排 Orchestrator + 多模型路由 + 自我进化 Memory"。

- **用户**：内部生产团队（1-3 人/部剧），后续 to B 提供给大规模内容生成团队
- **内容类型**：漫剧、3D 动画、真人 AI 剧，多风格切换
- **内容规模**：中篇为主（30-80 集，每集 1-2 分钟）
- **MVP 范围**：剧本 → 分镜 → 图片 → 视频（无声）
- **部署模式**：先本地部署，验证后上云

## 二、整体架构

四层架构：编排层 / Agent 层 / 基础设施层 / 存储层。

```
═══════ 用户交互层 ═══════
Web UI（审核/调整） │ CLI（批量任务） │ API（未来 to B）
          │
    ┌─────┴─────┐
    │ 编排层     │  Orchestrator Agent
    │ 受限决策   │  默认流程 + Memory 优化
    └─────┬─────┘
          │ 调度指令
    ┌─────┴──────────────────────────────┐
    │ Agent 层 — 专职 Agent 集群          │
    │                                     │
    │  📖 故事理解    🎬 分镜    👤 角色管理 │
    │  🎨 图片生成    🎥 视频生成  ✅ 质检   │
    └─────┬──────────────────────────────┘
          │ API 调用
    ┌─────┴──────────────────────────────┐
    │ 基础设施层                          │
    │                                     │
    │  🔥 爆款策略引擎  🔀 模型路由        │
    │  📋 任务队列      🧠 Memory         │
    └─────┬──────────────────────────────┘
          │ 持久化
    ┌─────┴──────────────────────────────┐
    │ 存储层                              │
    │ 项目数据 │ 资产库 │ 策略库 │ Memory │ 产物 │
    └────────────────────────────────────┘
```

### 核心设计原则

1. **编排 Agent 有"默认剧本"** — 默认按 故事理解→角色资产→分镜→图片→视频 执行，可根据 Memory 微调每步的参数和策略，但不能跳过核心阶段。
2. **Agent 之间通过结构化数据通信** — JSON schema 定义的数据结构，不是自然语言传话。
3. **质检 Agent 是闭环的关键** — 反馈回编排 Agent，形成"生成→评估→重试"循环。
4. **模型路由是独立服务** — 所有生成类 Agent 通过它选模型，不直接调用模型 API。

## 三、默认执行流程

### 阶段 0：用户输入

- 输入：剧本文本（或大纲）+ 风格指定（漫剧/3D/真人）+ 集数 + 特殊要求
- 编排 Agent 接收，启动默认流程

### 阶段 1：故事理解（串行）

- **Agent**：故事理解 Agent
- **输入**：完整剧本文本
- **工作**：解析世界观、人物关系图、每集情感弧、叙事节拍、每场戏的场景标签。标注每集的叙事角色（钩子集/付费卡点集/高潮集/过渡集）。
- **输出**：故事蓝图（Story Blueprint）— 全剧的"上帝视角"结构化数据
- **人工检查点**：蓝图生成后可暂停，供人审核角色设定和剧情理解是否准确

### 阶段 2：角色资产生成（串行）

- **Agent**：角色管理 Agent
- **输入**：故事蓝图中的角色描述 + 风格指定
- **工作**：为每个角色生成三视图/参考图，标记各阶段的外观状态变化。在全局资产库中检索可复用资产，缺失的新建。
- **输出**：角色资产档案 — 每个角色的视觉参考 + 状态时间线
- **人工检查点**：角色形象必须确认，这是后续所有画面的基准，此时修改成本最低

### 阶段 3：分镜生成（串行）

- **Agent**：分镜 Agent + 爆款策略引擎
- **输入**：故事蓝图 + 角色资产档案
- **工作**：
  1. 策略引擎规则粗筛 → 5~8 个候选策略
  2. 分镜 Agent（LLM）结合上下文精选策略 + 生成结构化分镜
  3. 每个分镜包含：景别、构图、运镜、情绪、角色引用、结构化提示词
  4. 分镜 Agent 标注每个镜头的集内节奏位置和内容类型
- **输出**：全剧分镜表 — 每集每个镜头的完整描述
- **人工检查点**：可选，审核关键集的分镜节奏

### 阶段 4：图片生成（可并行）

- **Agent**：图片生成 Agent × N（并行）
- **输入**：单个分镜的结构化提示词 + 对应角色资产
- **工作**：
  1. 编排 Agent 根据三维度权重计算综合重要性得分
  2. 模型路由根据得分 + 场景类型选模型
  3. 注入角色参考图（融图）保证一致性
  4. 质检 Agent 自动评估，不合格触发重试（重试次数由得分等级决定）
- **输出**：全剧静态画面
- **并行策略**：同一角色的锚点镜头（首次出场、重大变化）优先生成，后续镜头基于锚点参考

### 阶段 5：视频生成（可并行）— MVP 终点

- **Agent**：视频生成 Agent × N（并行）
- **输入**：静态画面 + 分镜中的运镜/动态指令
- **工作**：
  1. 模型路由根据动态复杂度选模型
  2. 质检 Agent 评估动态效果 + 角色一致性
- **输出**：全剧视频片段

### 阶段 6（后续版本）

配音（TTS）→ 口型同步 → 背景音乐 → 字幕 → 剪辑合成 → 导出

### 编排 Agent 的决策边界

| 可以做 | 不能做 |
|-------|--------|
| 根据 Memory 调整某阶段的模型选择、参数、重试次数 | 跳过故事理解阶段直接生成分镜 |
| 某镜头质检不过时决定换模型重试还是降级通过 | 跳过角色资产生成直接出图 |
| 根据历史经验调整并行度 | 不经质检直接输出成品 |

## 四、资产库

### 双层结构

- **全局资产库**：跨项目共享，团队级别。带标签和搜索，方便检索复用。后续 to B 时可按团队/租户隔离。
- **项目资产库**：单部剧专属。从全局导入 + 项目内新建。项目结束后可"发布"优质资产到全局。

### 六类资产

| 资产类型 | 内容 | 关联关系 |
|---------|------|---------|
| 角色 | 三视图/表情包/参考图、状态时间线（服饰/年龄/造型变化）、性格标签 | 故事蓝图人物关系图 |
| 场景 | 背景参考图/全景图、光照条件、空间布局描述 | 分镜中的 setting 标签 |
| 道具 | 关键道具参考图/多角度、道具状态（完好/损坏/变形）、与角色的绑定关系 | 剧情中的道具出现时间线 |
| 服饰 | 各套服装参考图、穿着时间段（第N集~第M集）、与角色状态的关联 | 角色状态时间线 |
| 风格模板 | 画风参考/色调方案、光影风格/线条风格、可复用的整套视觉预设 | 模型路由的风格参数 |
| 音频 | 音效库/背景音乐/配乐、角色音色参考（TTS 参数）、按情绪/场景分类标签 | MVP 后启用，先建数据结构 |

### 资产在流水线中的流转

1. **故事理解阶段**：识别需要哪些资产 → 在全局库中检索已有匹配
2. **角色资产阶段**：全局库有 → 导入项目库；没有 → 新建并存入项目库
3. **分镜阶段**：每个镜头自动引用项目库中的角色 + 场景 + 道具 + 服饰资产
4. **图片生成阶段**：将引用的资产作为参考图注入生成模型（融图）
5. **视频生成阶段**：继承图片阶段的资产约束，保持一致性
6. **项目结束**：优质资产可"发布"到全局库，供后续项目复用

## 五、模型路由

### 路由输入：三维度重要性评分

镜头的综合重要性 = 剧集位置权重 × 集内节奏权重 × 内容类型权重。

#### 维度一：剧集位置策略

| 剧集类型 | 权重 | 说明 |
|---------|------|------|
| 钩子集（前 3~5 集） | ×1.5 | 决定用户是否继续看/是否付费，画面质量必须拉满 |
| 付费卡点集（通常第 8~12 集） | ×1.3 | 免费转付费的关键集，悬念和画面冲击力要最强 |
| 高潮集 / 大结局 | ×1.2 | 剧情转折、终极对决，关系到口碑和续集期待 |
| 过渡集 | ×1.0 | 铺垫、日常、支线叙事，质量达标即可 |

标注者：故事理解 Agent 在构建故事蓝图时标注。

#### 维度二：集内节奏策略

| 位置 | 权重 | 说明 |
|-----|------|------|
| 开场钩子（前 3~5 秒） | ×1.4 | 决定用户是否划走，第一个画面必须抓眼球 |
| 情绪高点 | ×1.2 | 反转/冲突爆发/感情释放，决定用户情绪记忆点 |
| 尾部钩子（最后 2~3 秒） | ×1.2 | 悬念/下集预告，决定用户是否点"下一集" |
| 中段叙事 | ×1.0 | 铺垫/过渡/日常 |

标注者：分镜 Agent 在生成分镜时标注。

#### 维度三：内容类型

| 类型 | 权重 | 说明 |
|-----|------|------|
| 角色首次出场 | ×1.3 | 定义角色视觉印象，后续一致性锚点 |
| 打斗 / 特效 / 大场面 | ×1.2 | 动态要求高 |
| 对话 / 特写 | ×1.0 | 重表情和一致性 |
| 空镜 / 环境 / 回忆 | ×0.8 | 一致性要求最低 |

标注者：分镜 Agent 根据镜头内容标注。

#### 综合得分示例

| 场景 | 计算 | 得分 | 等级 |
|-----|------|------|------|
| 第1集开场打斗 | 1.5 × 1.4 × 1.2 | 2.52 | S 极重要 |
| 第10集（付费卡点）尾部悬念 | 1.3 × 1.2 × 1.0 | 1.56 | A 重要 |
| 第25集中段日常对话 | 1.0 × 1.0 × 1.0 | 1.0 | B 普通 |
| 第40集中段空镜过渡 | 1.0 × 1.0 × 0.8 | 0.8 | C 填充 |

#### 得分 → 资源分配映射

| 得分区间 | 等级 | 模型选择 | 最大重试 | 预算占比 |
|---------|------|---------|---------|---------|
| ≥ 2.0 | S 极重要 | 最高质量模型 | 3 次 + 可换模型 | 不限 |
| 1.4 ~ 2.0 | A 重要 | 高质量模型 | 2 次 | 单集预算 20% |
| 1.0 ~ 1.4 | B 普通 | 性价比模型 | 1 次 | 按均值分配 |
| < 1.0 | C 填充 | 最经济模型 | 不重试 | 压缩到最低 |

人可以在审核分镜时手动覆盖某个镜头的重要性等级。

### 路由逻辑：两阶段

1. **规则筛选**：根据硬约束缩小范围（风格兼容性、能力要求、预算等级）→ 从全部模型中筛出可用候选集
2. **策略选择**：在候选集中根据重要性等级 + Memory 加权（历史质检通过率、API 稳定性）+ 连续性偏好（同集尽量同模型）做最终选择

### 模型适配器层（Provider Adapter）

每个模型 API 一个适配器，实现统一的 `ModelAdapter` 接口：

- `Capabilities()` — 声明能力（支持的风格、动态能力、口型同步、最大分辨率等）
- `Generate(prompt, assets, params)` — 统一生成接口
- `EstimateCost(params)` — 成本估算
- `HealthCheck()` — 健康检查

**接入新模型 = 实现 ModelAdapter 接口**，不改业务逻辑。适配器声明自己的能力，路由层根据能力声明做匹配。

### 成本控制

- **预算分配**：项目总预算 → 按集分配 → 每集内按镜头重要性加权
- **实时追踪**：每次 API 调用记录实际成本，按项目/集/镜头粒度汇总。超预算时编排 Agent 自动降级
- **重试经济学**：重试次数由重要性等级决定，见上表

## 六、爆款策略引擎

灵感来源：火山剧创的"200+ 爆款镜头策略库"。

### 策略数据结构

每条策略是带多维度标签的结构化数据：

- `id` — 策略唯一标识
- `name` — 策略名称
- `tags` — 适用场景标签（narrative_beat, emotion_arc, setting, pacing, character_count 等）
- `shot_formula` — 镜头公式（景别、构图、运镜、时长等结构化参数）
- `examples` — 示例描述或参考
- `weight` — 有效性权重（初始 1.0，运营数据回流后更新）

### 两阶段匹配

**阶段一：规则粗筛**
- 输入：故事理解 Agent 输出的场景标签
- 过程：标签匹配，200+ 策略中筛出 5~8 个候选
- 特点：快速、确定性、零 LLM 成本

**阶段二：LLM 精选**
- 输入：5~8 个候选策略（含镜头公式 + 示例）+ 当前场景完整剧本上下文 + 前 2~3 个镜头已选策略（防重复）+ 整集情绪节奏曲线
- 过程：LLM 综合考虑上下文、情绪衔接、审美多样性做最终决策
- 输出：最终选定 1~2 个策略 + 微调建议

### 进化机制

运营数据回流后，根据使用某策略的剧集的播放量/完播率/付费转化更新策略权重。高效策略提权，低效策略降权但不删除。

## 七、Memory 系统

### 三层结构

#### 短期 Memory

- **范围**：单次任务 / 单部剧生产过程
- **内容**：当前项目的角色资产引用关系、已生成镜头的视觉参考（锚点）、本次质检反馈和重试记录、编排 Agent 的决策链路
- **生命周期**：任务结束后归档
- **作用**：保证单部剧生产过程中的上下文连贯

#### 中期 Memory

- **范围**：跨项目，近期经验（30~90 天）
- **内容**：模型表现统计（各模型在不同场景的质检通过率）、参数组合效果、策略命中记录、常见失败模式
- **生命周期**：滑动窗口，过期数据降权但不删除
- **作用**：优化模型路由、策略选择、参数调优

#### 长期 Memory

- **范围**：永久积累，团队级知识
- **内容**：运营数据关联（视觉风格/节奏/策略 vs 播放/转化）、行业模式沉淀、模型演变记录、团队偏好（人工修改最多的环节和方向）
- **生命周期**：永久，定期压缩和提炼
- **作用**：指导内容策略、长期优化方向
- **to B 扩展**：各客户独立长期 Memory

### 写入时机

| 触发时机 | 写入者 | 写入内容 | Memory 层 |
|---------|-------|---------|----------|
| 每次模型调用完成 | 模型路由 | 模型 + 参数 + 耗时 + 成本 | 短期 |
| 质检 Agent 评估完成 | 质检 Agent | 通过/不通过 + 扣分项 + 对应模型和参数 | 短期 → 聚合到中期 |
| 编排 Agent 做决策时 | 编排 Agent | 决策上下文 + 选择 + 结果（事后补） | 短期 → 聚合到中期 |
| 人工审核/修改时 | 系统记录 | 人改了什么、改之前是什么 → 隐含偏好 | 中期 → 提炼到长期 |
| 项目完成时 | 编排 Agent | 项目总结：成本、耗时、质量评分、关键决策回顾 | 中期 |
| 运营数据回流时 | 数据管道 | 播放量/完播率/付费转化 关联到具体剧/集/策略 | 长期 |

### 读取场景

- **编排 Agent**：任务开始时读中期 Memory 了解近期模型表现；每次决策前读短期 Memory 了解本任务进展；决定重试策略时读历史重试成功率
- **模型路由**：选模型时读中期 Memory 的模型表现统计；新模型上线时读长期 Memory 判断适用性
- **策略引擎**：匹配时读长期 Memory 获取策略有效性权重
- **各专职 Agent**：读历史上效果好的 prompt 模板、常见问题、一致性经验

### 进化飞轮

```
生产更多剧 → 积累更多 Memory → 决策更精准
→ 质量更高 / 成本更低 → 运营数据更好
→ 反馈回 Memory → 策略权重更准 → 下一部剧更好
```

### MVP 范围

MVP 阶段只做短期 + 中期 Memory。长期 Memory（运营数据关联）在运营数据管道打通后启用。但数据结构从 MVP 就预留。

## 八、技术栈

### Go + TypeScript 双语言架构

| 系统层 | 技术选型 | 选型理由 |
|-------|---------|---------|
| 用户交互层 | TypeScript + React/Next.js + TailwindCSS/shadcn | 审核界面需要丰富交互（分镜拖拽、画面预览、对比） |
| API 网关 | Go + Gin/Echo + WebSocket | 高并发 + 实时进度推送 |
| 编排 Agent | Go — 自建编排引擎，interface 定义 Agent 协议 | 不依赖 LangGraph 等重框架，完全可控 |
| 专职 Agent | Go — 每个 Agent 一个 package，统一 Agent 接口 | 核心逻辑：拼 prompt → 调 LLM API → 解析输出。强类型保证数据结构一致 |
| 策略引擎 | Go — 标签匹配 + LLM 调用 | 和 Agent 层共享 SDK |
| 模型路由 | Go — Adapter 模式 | interface 天然适合 Adapter 模式，新模型 = 实现接口 |
| 任务队列 | MVP: goroutine + channel → 上云: Asynq + Redis | 渐进式升级 |
| Memory 存储 | MVP: SQLite → 上云: PostgreSQL + pgvector | 结构化查询 + 向量检索一个库覆盖 |
| 文件存储 | MVP: 本地文件系统 → 上云: MinIO/S3 | S3 兼容 API，平滑迁移 |

### Go Agent 核心模式

```go
type Agent interface {
    // 声明能力
    Capabilities() AgentCapabilities
    // 核心执行
    Run(ctx context.Context, input AgentInput) (AgentOutput, error)
    // 健康检查
    HealthCheck() error
}

// 每个专职 Agent 实现这个接口
// StoryAgent, StoryboardAgent, ImageGenAgent, VideoGenAgent,
// CharacterAgent, QualityCheckAgent
```

### MVP 简化策略

| 组件 | MVP | 上云 / to B |
|-----|-----|------------|
| 任务调度 | goroutine + channel | Asynq + Redis |
| 文件存储 | 本地文件系统 | MinIO / S3 |
| 数据库 | SQLite | PostgreSQL + pgvector |
| 前端 | 简单 React 审核界面 | Next.js + 多租户 |
| Memory | SQLite 表 + JSON 字段 | PostgreSQL + 向量检索 |

MVP 阶段零外部依赖，`go run` 即可启动。

## 九、质检 Agent 评估维度

质检 Agent 在图片生成和视频生成阶段介入，评估维度如下：

| 维度 | 权重 | 评估内容 |
|-----|------|---------|
| 角色一致性 | 30% | 与角色资产档案的视觉匹配度（面部特征、服饰、体型） |
| 画面质量 | 25% | 清晰度、伪影、畸变、手部/面部异常 |
| 分镜还原度 | 20% | 景别/构图/运镜是否符合分镜描述 |
| 风格一致性 | 15% | 与项目风格模板的匹配度，与同集前后镜头的风格连贯 |
| 叙事准确性 | 10% | 画面内容是否准确表达剧本意图（角色数量、动作、表情） |

评估方式：LLM 多模态能力（视觉理解），输入生成画面 + 分镜描述 + 角色资产参考图，输出结构化评分 + 具体扣分项。

通过阈值由重要性等级决定：
- S 级：≥ 85 分通过
- A 级：≥ 75 分通过
- B 级：≥ 65 分通过
- C 级：≥ 55 分通过

## 十、MVP 交付物定义

1. **能跑通的全链路**：输入一个剧本文本，输出全剧的无声视频片段
2. **编排 Agent + 6 个专职 Agent**：故事理解、角色管理、分镜、图片生成、视频生成、质检
3. **模型路由**：至少接入 2-3 个图片生成模型 + 2-3 个视频生成模型，支持三维度重要性路由
4. **爆款策略引擎**：初始策略库（可从行业案例提炼），两阶段匹配跑通
5. **资产库**：双层结构（全局 + 项目），六类资产数据结构就绪，MVP 主要使用角色 + 场景 + 风格模板
6. **短期 + 中期 Memory**：记录模型表现、质检结果、决策链路，跨项目积累
7. **Web UI**：分镜审核、角色确认、生成进度查看、质量对比
8. **人工检查点**：故事蓝图审核、角色形象确认、分镜审核（可选）
